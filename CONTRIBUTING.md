# Contributing

愿意花时间看这里，本身就很感谢。

这个项目最初是围绕真实茶叶生意场景做出来的，所以我比较在意两件事：

- 功能是不是真的顺手
- 改动会不会把现有流程弄复杂

如果你愿意提建议、修 bug、补文档，都是很欢迎的。

## 开始之前

先了解一下项目用了什么：

| 模块 | 技术 |
| --- | --- |
| 后端 (`packages/server`) | NestJS 10、TypeORM、SQLite (sql.js)、JWT |
| Web 端 (`packages/web`) | React 18、Vite、Ant Design 5、ProComponents、Zustand |
| 移动端 (`packages/mobile`) | React 18、Vite、PWA、Zustand |

然后把 README 跑一遍，确认本地环境能正常启动：

```bash
pnpm install
cp packages/server/.env.example packages/server/.env
pnpm dev:all
```

常用构建命令：

```bash
pnpm build:server
pnpm build:web
pnpm --filter mobile build
```

> 注意：项目目前没有配置自动化测试，改完后请手动跑一下相关流程确认没问题。

## 我比较推荐的提交流程

1. 先开 Issue 或 Discussion，简单说一下你准备改什么
2. 从 `main` 拉一个新分支，命名建议：`feat/xxx`、`fix/xxx`、`docs/xxx`
3. 小步提交，尽量别一口气塞太多不相关内容
4. 改完后自己先跑一遍相关构建和基本流程
5. 提 PR 时把改动背景、取舍和影响范围写清楚，PR 目标分支为 `main`

如果只是很小的 typo、文案、注释修正，也可以直接提 PR，不用太拘谨。

## 提交代码时请注意

- 不要提交真实密钥、令牌、账号、数据库文件
- 不要把本地产物、调试日志、截图缓存之类一并带上来
- 涉及 UI 的改动，最好附一两张截图
- 涉及流程调整的改动，最好写清楚“原来怎样、现在怎样”

## 关于代码风格

这个仓库没有特别重的流程，但希望大家尽量保持几件事：

- 命名直接一点，别故作高级
- 注释写给后来的人看，少写废话
- 看到可以顺手补的 README / 文档，欢迎一起补上
- 改动尽量贴近现有结构，不要为了“更优雅”把整个目录翻一遍

## Commit 说明

这个仓库的 commit message 约定是中文，并遵循下面这种格式：

```text
feat(web): 新增销售单打印备注
fix(mobile): 修复语音识别首字丢失
docs: 补充移动端语音配置说明
```

简单说就是：

- 前缀用 `feat` / `fix` / `docs` 这类常见类型
- scope 可选
- 冒号后面尽量用中文，写清楚这次到底改了什么

## 哪些改动我会特别欢迎

- 真正解决业务场景的 bug
- 减少误操作、提升录单效率的优化
- 文档补全，尤其是部署、配置、排错相关
- 移动端体验优化

## 哪些改动最好先沟通

- 大范围重构
- 数据结构或接口协议调整
- 权限模型变化
- AI 能力接入方式变化
- Docker / 部署链路重做

不是不能改，而是这类改动通常牵一发而动全身，先对齐一下会省很多时间。

## 最后

如果你的 PR 最终没有被合并，也不代表改动没有价值。很多时候只是时机、方向或者兼容成本的问题。

愿意认真提问题、认真写清楚背景的人，我都很感谢。
